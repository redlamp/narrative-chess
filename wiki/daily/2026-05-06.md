## Hydration mismatch fix on Clock + tickNow (PR #28)

Manual smoke on Vercel dev preview surfaced a Next.js console error
when a timed game ran. React reported a hydration mismatch on `Clock`'s
`data-displayed-ms` attribute (and the rendered text). Diff:

```
- data-displayed-ms="86382035"  (server)
+ data-displayed-ms={86381816}  (client)
```

### Root cause

Both `Clock.tsx` and `GameClient.tsx` seeded a clock-tick state via
`useState<number>(() => Date.now())`. The initializer runs once at SSR
(with the server's wall clock) and once at hydrate (with the client's),
producing different values for the first paint. React flags + refuses
to patch.

### Fix

Initialize both states to `null`. First render (SSR + initial hydrate)
uses raw `remainingMs` (Clock) / treats `myClockExpired` as `false`
(GameClient). Mount effect seeds the actual `Date.now()` post-hydrate;
interval keeps it fresh.

- `Clock.tsx`: `now` init → `null`. When `now === null`, `displayed = remainingMs` (no interpolation). Tick interval skips while null.
- `GameClient.tsx`: `tickNow` init → `null`. `myClockExpired` short-circuits to `false` while null. Mount effect seeds `tickNow = Date.now()` even outside `in_progress` so the guard activates immediately post-hydrate.

PR #28 → CI green → merged via `--merge` (`--no-ff`). Merge commit
`99d4890`. Branch deleted local + remote.

### Why this didn't show up earlier

E2E spec runs in headless Chromium against `localhost:3000` dev server.
Dev mode in Next 16 uses Turbopack but the page hydrates the same way.
Possible explanations:

1. The e2e was reading `data-side` and `data-active` attrs but not
   `data-displayed-ms`, so didn't fail on the mismatch (React still
   logs to console but doesn't throw).
2. CI Playwright job is currently no-op (task #13 — `compgen` glob
   bug), so the spec never ran in CI to catch it pre-deploy.
3. Local-dev didn't see it because the user only smoked on Vercel
   preview, where SSR + edge-cached HTML widens the now-deltas.

(2) is fixable; raising priority of task #13.

## Pattern memo

Cross-project takeaway already added to
`~/.claude/memory/tools/react-19-lint-patterns.md` last session.
Updating the same file to add the SSR hydration variant: when a
component's render touches `Date.now()` (directly or via `useState`
initializer), the value diverges between server and client. Pattern
to use:

```tsx
const [now, setNow] = useState<number | null>(null);
useEffect(() => { setNow(Date.now()); /* + interval */ }, [...]);
const displayed = now === null ? fallback : computeFromNow(now);
```

Server and client both render `fallback` on the first frame; client
swaps to the live computation post-hydrate.

## PR #29 — open-challenges visibility (NULL-side filter)

User reported that an open game on account A wasn't visible in
account B's "Other players' open challenges" section on `/games`.

Root cause: `app/games/page.tsx` used
`.not("white_id", "eq", uid).not("black_id", "eq", uid)` to exclude
games the viewer was a participant in. PostgREST translates `.not.eq`
to `NOT (col = uid)`. Against NULL that evaluates to NULL (treated as
false in WHERE), so every open game (which by design has one NULL
side) was silently excluded.

Fix: NULL-aware OR groups so NULL counts as "not the viewer":

```ts
.or(`white_id.is.null,white_id.neq.${uid}`)
.or(`black_id.is.null,black_id.neq.${uid}`)
```

Verified against the live row (white_id=NULL, black_id=Alice). Bob's
query now passes both OR groups; row appears in his directory under
"Other players' open challenges." Alice still sees it under
"Your open challenges" via the existing `or(white_id.eq.uid,
black_id.eq.uid)` query.

Merged via PR #29 at merge commit `91591d9`. Branch
`fix/open-challenges-null-side` deleted local + remote.

### Pattern memo (cross-project)

PostgREST `.not.eq` against nullable columns is a foot-gun. SQL
three-valued logic means `NOT (NULL = x)` is NULL, not true. Any
filter that needs "exclude rows where col matches X" against a
nullable column should be written as
`(col is null) OR (col neq X)` to handle NULL explicitly.

## PR #31 — live games lobby + game-started toast

User reported that the games-directory `/games` page didn't update
when an open game flipped to `in_progress` (account A on `/games`,
account B joins via game URL → A's list stayed stale until
refresh). Also wanted a "game started" notification.

Fix:

- `lib/realtime/subscribe.ts` gains `subscribeToAllGameStatus()` —
  subscribes to UPDATE + INSERT events on the `games` table without
  an id filter. RLS handles delivery: participants get all their
  games' updates; non-participants only receive `open` rows.
- `app/games/GamesRealtime.tsx` (new) client component mounted at
  the top of `/games`. Per-id `lastStatus` ref filters out
  ply/fen-only updates within an in-progress game (those don't
  change status, so no refresh fires). Per-id `toasted` set
  deduplicates the open→in_progress toast. On any observed status
  change, calls `router.refresh()` so the four lists rebuild from
  server data.

Caveat (acceptable v1, documented in PR): cross-account "challenge
disappears" stays best-effort. Bob has Alice's open game in his
list; Alice gets a joiner; her row flips to `in_progress`. Bob's
RLS no longer grants delivery on the `in_progress` row, so his
list stays stale until next page load. 30s polling fallback
deferred until/unless this surfaces in practice.

Merged via PR #31 at merge commit `f278824`. Both branches deleted.
